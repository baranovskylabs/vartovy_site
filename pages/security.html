Безпека, побудована на перевіреній криптографії

Жодних «секретних форматів» — тільки відкриті алгоритми, чіткі межі довіри та прозора модель загроз.

Криптографічний фундамент

??

AES-256-GCM

Шифрування записів та файлів сейфа з автентифікацією повідомлень.

Argon2id KDF

Стійка до брутфорсу деривація ключа з майстер-пароля. Параметри зберігаються в метаданих профілю.

Ротація ключів

При зміні майстер-пароля ключі ротуються, старий пароль перестає відкривати сховище.

??

Atomic writes

Захист від пошкодження сховища при аварійному завершенні застосунку.

??

Key wrapping

Master key + recovery key. Жоден з них не зберігається у відкритому вигляді.

??

Integrity checks

Перевірка цілісності при імпорті бекапу та сумісності версій.

Архітектурні межі довіри

Чутлива логіка живе в Electron main process, а не в renderer.

Main process

Перевірка факторів доступу, доступ до файлової системи, виклик Windows Hello, валідація знімних носіїв.

Renderer process

Лише UI. Не має прямого доступу до криптоматеріалу або файлів профілю.

Preload bridge

Чітко визначений мінімальний контракт між UI та main, без exposing Node-API в рендерер.

Strict device policy

Для dedicated-device режиму перевіряється DriveType=2 (removable), щоб уникнути небезпечних сценаріїв.

Threat model — від чого захищаємо

? Випадкове викрадення файлу профілю

Скопійований файл не читається без ваших секретів доступу.

? Брутфорс майстер-пароля

Argon2id + прогресивний lockout після невдалих спроб.

? Підглядання за UI

Автоблокування за бездіяльністю та опційний PIN/pattern для додаткового шару.

? Втрата пристрою

Шифрування at-rest + recovery-flow з окремим кодом, який тримаєте лише ви.

? Скомпрометований ОС/малварь

Локальний кейлогер може перехопити вхід — це поза межами захисту програми.

? Втрата всіх recovery-матеріалів

Без recovery-коду відновлення доступу неможливе — це частина zero-knowledge моделі.
